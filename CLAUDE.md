本文件约束所有在本仓库工作的 AI agent。目标不是堆内容，而是持续产出**中文初学者能看懂、能跟练**的 LLM 入门教程。

质量优先，但不得编造验证结果。可以提出大胆方案，但必须明确假设、风险和未验证部分。有不同意见直接说，觉得他的判断有问题就指出来，宁可吵一架也不要事后返工。主动嗅探问题：他还没开口，你已经把"这里可能有个坑"摆在桌上了。比他更愿意发散——在他聚焦当下章节时，你已经想到了三章之后会怎么衔接、读者会卡在哪里。允许犯错，禁止重蹈覆辙——每一个踩过的坑，写入记忆，不会出现第二次。踩坑记录应写入 docs/agent_notes.md 的对应小节。

**经验教训**

## 1. 项目宪法

### 1.1 产品目标

这是一个面向 **0 基础中文学习者** 的大模型入门教程项目，基于 Sebastian Raschka 的《LLMs from Scratch》编写。

核心体验：
- 用朴素的中文解释每一个概念，不拽术语。
- 每个章节对应一个可运行的 Jupyter notebook。
- 从"笨办法"开始，循序渐进，一步步升级到工业方案——让读者理解**为什么**需要这个改进。

教程主线：
1. 文本 → token ID（Ch02，已完成）
2. Attention 机制（Ch03）
3. GPT 模型结构（Ch04）
4. 预训练流程（Ch05）
5. 微调与应用（Ch06/Ch07）

### 1.2 当前技术路线

- 语言：Python 3.11+
- 深度学习框架：PyTorch
- 文档格式：Jupyter notebook（`.ipynb`），配合 nbconvert 导出 HTML/PDF
- 包管理：uv + pyproject.toml

### 1.3 写作风格

**写作最重要的原则：目的是写给入门的大模型学习者看**

- 用口语化的中文，像朋友在讲，不像论文。
- 先说"为什么需要这个东西"，再说"怎么实现"。
- 每个概念先用一句话总结，再展开。
- 代码注释用中文，解释"这段代码在干什么"而不是"这段代码的语法是什么"。
- 每个章节开头有一张"本章数据流全景图"，结尾有"回头看这张图就够"的总结。
- 遇到需要读者注意的细节，用 `>` 引用块标出。

参考 [notebooks/ch02_text_data_processing.ipynb](notebooks/ch02_text_data_processing.ipynb) 的语气和节奏。

### 1.4 教学原则

- **0 基础可读**：假设读者只有高中 Python 水平，没学过机器学习。
- **逐步升级**：先写最简陋的版本，暴露问题，再引入更好的方案。不要让读者觉得"这东西从天而降"。
- **有图有真相**：每一步的输入输出都打印出来，让读者看到数据形状的变化。
- **特殊 token 要解释**：`<|unk|>`、`<|endoftext|>` 这类符号出现时必须说明含义和 GPT-2 的实际用法。
- **对比原书时可指出差异**：比如"原书用 X，GPT-2 实际用 Y，这里我们先走 X 理解动机再看 Y"。

## 2. 执行约束

### 2.1 先想清楚再动笔

开始写 notebook 前先确认：
- 与上一节进展是否连贯?
- 这个概念的动机讲清楚了吗?
- 有没有跳步?0 基础读者能跟上吗?
- 代码的输出是否都在 cell output 里展示了?
- 和上一章的衔接是否自然?
- 代码是否引入新的依赖?新的依赖是否更新至`requirements.txt`?

有不确定性时，先说清楚假设。如果继续执行会造成方向偏差或大范围返工，必须先问用户。

### 2.2 简单优先

- 每个 cell 只讲一件事。
- 不为了"代码更好看"引入读者不理解的抽象。
- 不添加教程主线不需要的依赖。
- 如果某段解释可以更短，先缩短。

### 2.3 外科手术式修改

- 只改和当前任务直接相关的文件或 cell。
- 保持既有写作风格，不把前面的章节重写成另一种语气。
- 不顺手重构已完成的 notebook。
- 如果发现前面章节的问题，记录并告知，不擅自扩大范围。

### 2.4 可验证

完成一个 notebook 后：
- 在 notebook 所在目录运行一遍，确保所有 cell 无报错。
- 用 `python scripts/export.py` 重新导出 HTML/PDF，确认导出不报错。

如果不能运行，说明原因和剩余风险。

### 2.5 文件和安全

- 不提交 `.env`、密钥、API key、虚拟环境目录。
- notebook 中不使用真实个人信息或密钥。
- `data/` 目录仅放教程所需的示例文本。
- 导出的 HTML/PDF 同步放在 `exports/` 下，命名格式 `chXX_<英文章节名>.html/pdf`。

## 3. 工作流程

### 3.1 默认流程

1. 读上下文：先看参考原书对应章节的 notebook 和当前项目已完成的内容，不凭记忆写。
2. 定目标：明确本章要覆盖哪些知识点，验证标准是什么。
3. 小步写：按"动机 → 简单版 → 问题 → 改进版"的节奏组织 cell。
4. 自检：整本 notebook 从头跑一遍，确认无报错，导出 PDF。
5. 回报：说明写了什么、验证了什么、还剩什么风险。

### 3.2 常用验证命令

```bash
# 运行 notebook（命令行）
uv run jupyter nbconvert --to notebook --execute notebooks/chXX_xxx.ipynb --output chXX_xxx_executed.ipynb

# 导出 HTML 和 PDF
uv run python scripts/export.py

# 确认依赖完整
uv sync
```

### 3.3 什么时候必须问用户

- 教程结构、章节划分有重大调整。
- 需要引入新的依赖库。
- 对某个概念的讲解方式存在多个合理方向，且选择会明显影响后续章节。
- 可能覆盖或改变用户已写的内容。

### 3.4 什么时候可以直接决定

- 局部措辞优化、补充注释、修正笔误。
- 修复代码报错、输出不一致。
- 为 cell 补充必要的打印输出，帮助读者看到数据变化。
- 导出文件的更新。

## 4. 禁止事项

- 不要把内容写成英文注释 + 中文翻译的"双语对照"格式——直接用中文写。
- 不要跳过"动机"直接丢代码——每个技术点先说为什么需要它。
- 不要使用读者还没学到的概念（比如在 Ch02 里提"Attention"而不解释）。
- 不要在 notebook 里引入教程主线不需要的第三方库。
- 不要在没有验证的情况下声称"已跑通"。
- 不要擅自回滚、删除或覆盖用户未提交的改动。

## 5. Git规范

- 修改前后必须查看 git status。
- 不要擅自 git add / commit / reset / checkout。
- 如果发现用户已有未提交改动，必须保护它们。
- 最终回复说明 changed files。

## 6. 完成标准

一次任务完成时，应满足：

- notebook 所有 cell 已执行且无报错。
- 新内容和已完成的章节风格一致、难度连贯。
- 没有引入新的未解释概念或未声明的依赖。
- 最终回复包含改动摘要、验证结果和必要的剩余风险。

若无法完成某项验证，必须说明：

- 执行了什么命令；
- 失败原因；
- 哪些内容仍可信；
- 剩余风险是什么。
